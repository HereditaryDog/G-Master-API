# Subscription Group Billing

本文档说明 G-Master API 中订阅升级分组和账户余额之间的扣费规则。

## 背景

订阅套餐可以配置 `upgrade_group`，用户购买后会临时升级到对应分组，例如周卡用户组或月卡用户组。此前系统更多依赖用户计费偏好判断订阅优先或钱包优先，容易出现用户无法明确控制本次请求扣哪一类额度的问题。

当前规则调整为：用户本次选择的分组决定扣费来源。

## 扣费规则

如果用户存在有效的订阅升级分组：

- 选择订阅套餐对应的 `upgrade_group` 时，扣订阅额度。
- 选择非订阅升级分组，例如 VIP 分组时，扣账户余额。
- 订阅额度不足时，不自动从账户余额补扣；用户需要切换到 VIP 等普通分组后再调用。

如果用户没有有效的订阅升级分组：

- 继续使用原有可用分组规则。
- 继续使用用户原有计费偏好配置。

## 可选分组规则

拥有有效订阅升级分组的用户，可以选择所有已配置分组。这样用户可以自己决定：

- 使用周卡/月卡分组，消耗订阅额度。
- 使用 VIP 或其他普通分组，消耗账户余额。

系统不会修改令牌已保存的分组，只会在请求时按令牌或操练场当前选择的分组执行路由和扣费。

## 相关代码

- `service/group.go`：计算订阅用户的可用分组。
- `middleware/auth.go`：令牌分组鉴权。
- `middleware/distributor.go`：操练场分组选择和渠道分配。
- `service/billing_session.go`：根据本次选择的分组绑定扣费来源。
- `model/subscription.go`：查询有效订阅升级分组。

## 验证建议

建议至少验证以下场景：

- 有效周卡用户选择周卡分组，请求扣订阅额度。
- 有效周卡用户选择 VIP 分组，请求扣账户余额。
- 周卡额度耗尽后，选择周卡分组返回订阅额度不足。
- 周卡额度耗尽后，选择 VIP 分组仍可使用账户余额调用。
- 无有效订阅用户无法越权选择未授权分组。
